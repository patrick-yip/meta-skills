# Contributing to meta-skills

Thanks for helping curate the meta-skills ecosystem. This list is opinionated by design. Not everything gets in.

## Submission Criteria

A skill/tool belongs here if it meets **all** of these:

1. **Meta-level** — It operates on skills themselves (creates, validates, discovers, distributes, analyzes, or migrates skills). Tools that happen to be skills but don't operate on the skill layer don't qualify.
2. **Public repo** — Must be publicly accessible on GitHub (or similar). No closed-source, no waitlists.
3. **Functional** — Must work. Archived repos, empty READMEs, or broken installs will be removed.
4. **Not malicious** — No history of malware, supply chain attacks, or obfuscated code. If a repo had an incident but addressed it transparently, note the incident in the editorial comment.

## How to Submit

1. **Open an issue first** — Describe the tool and why it belongs. This saves everyone time if it's out of scope.
2. **One tool per PR** — Makes review manageable.
3. **Follow the entry format:**

```
- **[tool-name](https://github.com/owner/repo)** — What it does. Why it's awesome or notable. ⭐ <star count>
```

4. **Place it in the right category** — Authoring, Quality, Discovery, Distribution, or Analysis. If unsure, mention it in the PR description.
5. **Include an editorial note** — Don't just describe what it does. Say why it matters, what makes it stand out, or what to watch out for. This is what separates us from a raw link dump.

## What Gets Rejected

- Tools that are just skills (not meta-skills)
- Self-promotion without substance (your tool needs to actually work and be useful)
- Duplicate functionality without meaningful differentiation
- Repos with zero commits beyond initial scaffold

## Updating Star Counts

Star counts are approximate and updated periodically. Don't PR just to update a number.

## Code of Conduct

Be constructive. Critique tools, not people. If you think a listed tool has quality or security issues, open an issue with specifics.
